# 基于 Commit 的版本管理

了解 Mut commit 模型如何替代传统的文件夹快照，为你的 Context File System 提供自动化的版本控制。

---

## 从快照到 Commit

早期版本的 PuppyOne 使用 `folder_snapshots` 表来保存文件夹级别的快照备份。当前版本已迁移到 **Mut commit** 模型：

- 每次写入操作自动创建一个版本化的 commit
- 版本历史存储在 `mut_commits` 表中
- 不再需要手动创建文件夹快照

> `folder_snapshots` 已废弃，不再在代码中使用。所有版本管理功能现在通过 Mut commit 实现。

---

## Mut Commit 模型

### 工作原理

```
写入 Content API
       │
       ▼
  创建 Mut commit
  (自动记录变更)
       │
       ▼
  存储到 mut_commits 表
  (project_id, path, version, content, author, timestamp)
```

每个 commit 记录了：

| 字段 | 说明 |
|------|------|
| `version` | 递增的版本号 |
| `path` | 被修改的文件路径 |
| `author` | 操作者（用户邮箱或 Agent 名称） |
| `created_at` | commit 时间 |
| `message` | 可选的提交信息 |

### 与文件夹快照的对比

| 维度 | 文件夹快照（已废弃） | Mut Commit |
|------|---------------------|------------|
| 触发方式 | 手动创建 | 每次写入自动创建 |
| 粒度 | 整个文件夹 | 单个文件 |
| 存储 | `folder_snapshots` 表 | `mut_commits` 表 |
| 回滚 | 整个目录回退 | 单文件精准回滚 |
| 开销 | 大（复制整个目录） | 小（仅记录变更） |

## 查看 Commit 历史

```bash
# 查看某个文件的 commit 历史
curl -X GET "https://api.puppyone.ai/api/v1/content/{project_id}/versions?path=/docs/product-spec.md" \
  -H "Authorization: Bearer {token}"
```

响应示例：

```json
{
  "versions": [
    {
      "version": 5,
      "created_at": "2025-03-09T15:30:00Z",
      "author": "Agent: data-sync",
      "size": 2048
    },
    {
      "version": 4,
      "created_at": "2025-03-09T14:00:00Z",
      "author": "user@example.com",
      "size": 1856
    }
  ]
}
```

---

## 回滚到指定版本

```bash
curl -X POST "https://api.puppyone.ai/api/v1/content/{project_id}/rollback" \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{"path": "/docs/product-spec.md", "version": 3}'
```

回滚本身也会创建一个新的 commit，历史永不丢失。

---

## 适用场景

| 场景 | 说明 |
|------|------|
| Agent 误写入 | 通过版本历史定位问题版本，回滚到正确版本 |
| 批量操作前的保护 | 每次写入都自动 commit，无需手动备份 |
| 数据迁移前存档 | 查看 commit 历史确认当前状态 |
| 追踪变更趋势 | 对比不同版本了解数据演变 |

---

## 注意事项

- 所有写入操作（Dashboard、API、MCP）都会自动创建 commit
- 版本号只增不减，回滚后会生成新的更高版本号
- 每个文件独立管理版本历史
- Commit 记录操作者信息，方便在[审计日志](/zh/audit)中追溯
