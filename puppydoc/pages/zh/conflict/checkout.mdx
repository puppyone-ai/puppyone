# MUT 并发控制模型

PuppyOne 使用 MUT（Mutation）引擎安全地处理并发写入，采用乐观并发控制和三方合并策略。

---

## 工作原理

PuppyOne 的 MUT 引擎不使用显式的检出/锁定机制，而是通过 **base_version 追踪** 和 **push 时三方合并** 来实现并发控制。这是一种乐观策略：多个操作者可以自由写入，冲突在写入时被检测和解决。

```
Agent A                         PuppyOne                        Agent B
  │                                │                               │
  ├── 读取文件 (获得 v3) ──────────▶│                               │
  │                                │                ├── 读取文件 (获得 v3)
  │   (本地编辑)                    │                │   (本地编辑)
  │                                │                               │
  ├── 写入 (base_version=3) ──────▶│                               │
  │                                │── 接受，创建 v4                │
  │◀── 成功 (v4) ─────────────────│                               │
  │                                │                               │
  │                                │      ├── 写入 (base_version=3)
  │                                │── 检测冲突 (base=3, current=4)
  │                                │── 尝试三方合并
  │                                │◀── 合并结果或冲突 ─────────────┤
```

---

## 带乐观并发的写入

通过 Content API 写入时，可以包含 `base_version` 来启用冲突检测：

### API

```bash
# 带 base_version 的乐观并发写入
curl -X POST "https://api.puppyone.ai/api/v1/content/{project_id}/write" \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{
    "path": "specs/api-design.md",
    "content": "# API Design\n\n更新了认证接口文档",
    "base_version": 3,
    "message": "更新认证接口文档"
  }'
```

如果文件在 v3 之后被修改过，MUT 引擎会：

1. **检测版本不匹配** — 当前版本比 base version 新
2. **尝试三方合并** — 比较基础版本 (v3)、当前版本和传入的修改
3. **返回结果** — 合并成功或返回需要解决的冲突详情

### 不使用 base_version

如果省略 `base_version`，写入将使用 last-write-wins 语义 — 直接覆盖当前内容。之前的版本保留在提交历史中。

---

## Push 时的三方合并

MUT 引擎的三方合并会比较三个版本：

```
        基础版本 (v3)
       ┌──────┴──────┐
       │             │
   当前版本 (v4)   传入的修改
    (Agent A)      (Agent B)
       │             │
       └──────┬──────┘
              │
        合并结果 (v5)
```

- 如果两个操作者修改了文件的**不同部分**，合并会自动成功
- 如果都修改了**同一部分**，返回冲突需要手动解决

---

## 处理合并冲突

当冲突无法自动解决时，API 会返回冲突详情：

```json
{
  "status": "conflict",
  "base_version": 3,
  "current_version": 4,
  "conflicts": [
    {
      "path": "/price",
      "base_value": 99,
      "current_value": 89,
      "incoming_value": 79
    }
  ]
}
```

解决方式：

1. **保留当前版本** — 放弃你的修改
2. **强制覆盖** — 不带 `base_version` 重新提交，使用 last-write-wins
3. **手动合并** — 合并双方修改后用最新的 `base_version` 提交

## 文件系统同步 (MUT 协议)

对于双向本地文件夹同步，MUT 协议在协议层处理并发：

- **Clone** — 下载项目的当前状态
- **Push** — 上传本地修改；自动应用三方合并
- **Pull** — 获取上次同步后的远程变更
- **Negotiate** — 交换状态以确定需要同步的内容

同步守护进程透明地使用此协议。Push 期间的冲突使用相同的三方合并逻辑解决。

---

## 注意事项

- **无显式锁** — 系统使用乐观并发控制，而非悲观锁
- **自动版本管理** — 每次成功写入都会在 `mut_commits` 中创建新的提交
- **完整历史** — 即使被覆盖的内容也会保留，可以回滚
- **审计追踪** — 每次写入操作都记录在审计日志中
