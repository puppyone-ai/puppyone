# MUT Scope 并发控制设计方案

> 状态：Draft — 待团队讨论  
> 日期：2026-04-14

---

## 一、背景

PuppyOne 的每个项目是一棵文件树。多个 Access Point（AP）通过 scope 机制划分各自的读写范围。
当多个 AP 并发写入同一项目时，需要保证：

- **不丢数据**：两个人改了不同文件，两边的改动都要保留
- **可见性**：子 scope 的改动，父 scope 能看到
- **高并发**：不相关的写入不应该互相阻塞

---

## 二、并发场景

下面列举所有可能的并发组合。先看完整张表，再总结规律。

### 场景列表

**同一 scope，同一 AP**

同一个客户端快速连续 push。实际中 CLI 是同步的，通常会等上一次完成再发下一次，
所以这更多是边界情况。但如果真的并发了，规则和下面一样。

| # | 场景 | 示例 | 期望行为 |
|---|------|------|---------|
| S1 | 改不同文件 | 第一次改 `a.md`，第二次改 `b.md` | 并行，两边改动都保留 |
| S2 | 改同一文件 | 两次都改 `readme.md` | 三方合并，冲突由服务端解决 |

**同一 scope，不同 AP**

两个 AP 都绑定了 `docs/` scope。

| # | 场景 | 示例 | 期望行为 |
|---|------|------|---------|
| S3 | 改不同文件 | AP-A 改 `docs/a.md`，AP-B 改 `docs/b.md` | 并行，两边改动都保留 |
| S4 | 改同一文件 | AP-A 和 AP-B 都改 `docs/readme.md` | 三方合并，冲突由服务端解决 |

**不同 scope，无嵌套**

AP-A 绑定 `docs/`，AP-B 绑定 `src/`，两个 scope 没有重叠。

| # | 场景 | 示例 | 期望行为 |
|---|------|------|---------|
| S5 | 各自 push | AP-A 改 `docs/a.md`，AP-B 改 `src/main.py` | 完全并行，互不影响 |

**不同 scope，有嵌套（父子）**

AP-A 绑定 root scope（`""`，覆盖整棵树），AP-B 绑定 `docs/`。
root scope 的写入范围包含了 `docs/` 的全部文件。

| # | 场景 | 示例 | 期望行为 |
|---|------|------|---------|
| S6 | 改不同文件 | AP-A 改 `config.json`，AP-B 改 `docs/readme.md` | 并行，两边改动都保留 |
| S7 | 改同一文件 | AP-A 改 `docs/readme.md`，AP-B 也改 `docs/readme.md` | 三方合并，冲突由服务端解决 |

**读取可见性**

| # | 场景 | 示例 | 期望行为 |
|---|------|------|---------|
| S8 | 子 scope 写入后，父 scope 读取 | `docs/` push 了新文件，root scope clone | 父 scope 必须能看到子 scope 的改动 |
| S9 | 父 scope 写入后，子 scope 读取 | root scope push 了 `docs/new.md`，`docs/` scope clone | 子 scope 必须能看到父 scope 在自己范围内的改动 |

### 规律总结

把上面 9 个场景放在一起看：

| 场景 | scope 关系 | 改的文件 | 期望行为 |
|------|-----------|---------|---------|
| **S1** | **同 scope 同 AP** | **不同文件** | **并行，都保留** |
| S2 | 同 scope 同 AP | 同一文件 | 三方合并 |
| **S3** | **同 scope 不同 AP** | **不同文件** | **并行，都保留** |
| S4 | 同 scope 不同 AP | 同一文件 | 三方合并 |
| **S5** | **不同 scope 无嵌套** | **不同文件** | **并行，互不影响** |
| **S6** | **嵌套 scope** | **不同文件** | **并行，都保留** |
| S7 | 嵌套 scope | 同一文件 | 三方合并 |
| S8 | 嵌套（子写父读） | — | 父 scope 可见 |
| S9 | 嵌套（父写子读） | — | 子 scope 可见 |

**结论：决定并发行为的唯一标准是"改没改同一个文件"。**

- S1、S3、S6 的 scope 关系完全不同（同 AP / 同 scope 不同 AP / 嵌套 scope），但期望行为一样 — 因为都是改不同文件
- S2、S4、S7 也是 scope 关系不同，但期望行为一样 — 因为都是改同一文件
- Scope 关系（同/异/嵌套）、AP 关系（同/异）都不影响并发行为，唯一的变量是文件

## 三、理想架构

### 3.1 单一项目树

整个项目维护一棵全局的 Merkle 文件树：

```
Project Root (root_hash)
├── docs/
│   ├── readme.md
│   └── guide.md
├── src/
│   └── main.py
└── config.json
```

- **一棵树，多个视图**：每个 scope 只是这棵全局树的某个子树，不是独立的孤岛
- **子 scope 写入自动嫁接回全局树**：`docs/` scope push 后，全局树自动更新，
  root scope 能看到 `docs/` 下的新文件
- **每个 scope 有自己独立的 hash 行**：CAS 只锁自己的行，不同 scope 天然并行

### 3.2 乐观并发（CAS + 自动重试）

写入流程：

```
1. 读取当前文件树状态（版本号 + hash）
2. 计算合并结果（server 端，不持锁）
3. 提交：检查"hash 是否还是我读的时候那个"
   - 是 → 写入成功
   - 否 → 说明有人在我之前提交了
         → 重新读取最新状态
         → 基于最新状态重新合并（三方合并自动保留不同文件的改动）
         → 再次尝试提交
```

**为什么选乐观并发，不选悲观锁？**

悲观锁（先抢锁，再写入）的问题：scope 粒度太粗。
如果 root scope 存在，它和所有子 scope 重叠，所有写入都要排队等 root scope 的锁。
**一个 root scope 的存在就会把整个项目的写入并发降到 1。**

乐观并发不加锁，大家并行计算合并，只有最后一步提交时才检查冲突。
改不同文件的 push 几乎永远不会真正冲突，重试一次就成功。

**CAS 的最后提交步骤是否也是串行的？**

是的，CAS 的"比较并写入"步骤本身是串行的 — 但它靠数据库行级锁保证原子性，
只持有**几毫秒**（一次 SQL UPDATE），不是几秒：

```sql
UPDATE mut_scope_state
SET version = 7, scope_hash = 'new_hash'
WHERE project_id = 'xxx'
  AND scope_path = 'docs/'
  AND scope_hash = 'old_hash'    -- CAS：只有 hash 没变才写入
-- affected rows = 1 → 成功
-- affected rows = 0 → 失败，触发重试
```

两个 push 同时执行这条 SQL 时，PostgreSQL 自动用行级锁串行化。
先到的成功，后到的失败并重试。

对比：

| | 串行范围 | 串行耗时 |
|---|---------|---------|
| 悲观锁 | 读文件 + 三方合并 + 写入 + 嫁接 + 版本号 | **几秒** |
| 乐观并发 | 只有最后的 CAS UPDATE | **几毫秒** |

乐观并发不是消除串行，而是把串行范围从整个 push（几秒）缩小到一次数据库原子操作（几毫秒）。
耗时的部分（读文件、计算合并）完全并行。

**极端并发下的瓶颈**

CAS 的吞吐上限约 200-500 次/秒（单行 UPDATE，取决于 DB 性能）。
如果同一个 scope 有上千并发 push：
- 几乎所有 CAS 都会首次失败
- 每次失败都要重新读 + 重新合并 + 再次 CAS → 重试风暴
- 吞吐急剧下降

但需要区分两种情况：

| 并发类型 | CAS 瓶颈？ | 原因 |
|---------|-----------|------|
| 1 万个 push 打到 1000 个不同 scope | **无** | 不同 scope 是不同 DB 行，天然并行 |
| 1000 个 push 打到同一 scope | **有** | 同一行反复 CAS 竞争 + 重试风暴 |

对于 PuppyOne 的场景（文件系统 + 多 Agent 协作），同一 scope 下几十个并发已是极端情况。
CAS 方案在这个量级完全够用。

如果未来需要支持同一 scope 的极端并发（几百+），可选方案：

- **写入队列**：同一 scope 的 push 排队，由 server 顺序处理，消除重试风暴
- **批量合并**：server 把短时间内到达的多个 push 合并成一次写入
- **分区（sharding）**：将大 scope 自动拆分为子 scope，分散写入压力

这些属于远期优化，当前阶段无需考虑。

**嵌套 scope 下的嫁接（graft）瓶颈**

前面说"不同 scope 是不同 DB 行，天然并行"。但子 scope push 成功后，
还需要把自己的新 hash 嫁接到父 scope 的树上 — 这步要更新父 scope 的行。

```
数据库行:
  (project_id, scope_path='',     scope_hash=...) ← root
  (project_id, scope_path='docs/', scope_hash=...) ← docs
  (project_id, scope_path='src/',  scope_hash=...) ← src
```

```
docs/ push:  CAS 更新 docs/ 行 ──→ 嫁接: CAS 更新 root 行
src/  push:  CAS 更新 src/  行 ──→ 嫁接: CAS 更新 root 行
                                          ↑
                                    所有子 scope 的嫁接
                                    都要 CAS 这同一行！
```

如果 root 下有 100 个子 scope 同时 push，它们各自的 scope CAS 互不影响（不同行），
但嫁接步骤都要 CAS root 行 → root 行变成热点 → 嫁接串行化。

**解法：把 push 和嫁接拆成两步**

| 步骤 | 操作 | CAS 目标 | 阻塞谁 |
|------|------|---------|--------|
| ① scope 提交 | CAS 更新自己的 scope_hash | 自己的行 | 只和同 scope 的其他 push 竞争 |
| ② 嫁接到父树 | CAS 更新父 scope 的 root_hash | 父 scope 的行 | 和兄弟 scope 的嫁接竞争 |

**步骤 ① 成功后立刻返回给 client**（数据已安全持久化）。
**步骤 ② 异步完成**（重试直到成功，通常几毫秒内完成）。

这样：
- 子 scope 的 push 响应时间不受兄弟 scope 影响
- 嫁接虽然串行，但只是快速的树操作（无文件 I/O），每次 ~1-2ms
- 唯一的代价：父 scope 看到子 scope 改动有**极短的延迟**（毫秒级），而非瞬时可见

这是**写入强一致 + 跨 scope 可见性最终一致**的模型：

| 保证 | 一致性级别 | 说明 |
|------|-----------|------|
| 子 scope 的数据不丢 | **强一致** | CAS 成功才返回 client |
| 父 scope 看到子 scope 改动 | **最终一致**（毫秒级） | 嫁接异步完成，极短延迟 |
| 同 scope 的并发合并 | **强一致** | CAS + 三方合并 |

### 3.3 Server 端完整流程（举例）

用一个具体例子说明 CAS 在 server 端实际发生了什么，以及 client 看到什么。

**场景**：Client A 改了 `a.md`，Client B 改了 `b.md`，同时 push。

```
初始状态：server v5, hash=X

Client A push (base=v5, 改了 a.md)
Client B push (base=v5, 改了 b.md)
                                        ┌─ Server 处理 ─┐
                                        │                │
Push A 到达 ─────────────────────────── ▶ 读 state: v5, hash=X
                                        │ 三方合并（无冲突）
                                        │ CAS: hash==X? ✅
                                        │ → 写入 v6, hash=A
                                        │ → 返回 Client A: 成功, v6
                                        │
Push B 到达 ─────────────────────────── ▶ 读 state: v5, hash=X
                                        │ 三方合并（无冲突）
                                        │ CAS: hash==X? ❌（已变成 A）
                                        │
                                        │ → 自动重试：
                                        │   读 state: v6, hash=A
                                        │   三方合并: base=v5, server=v6, client=B
                                        │   合并结果: a.md(A的) + b.md(B的) 都保留
                                        │   CAS: hash==A? ✅
                                        │   → 写入 v7, hash=B'
                                        │   → 返回 Client B: 成功, v7
                                        └─────────────────┘
```

**关键：CAS 重试完全在 server 端发生，client 无感知。** Client B 只发了一次 push 请求，等到响应"成功, v7"。

### 3.4 Client 端版本一致性

CAS 解决了 server 端的数据安全，但带来了一个新问题：**push 成功后，client 的本地状态可能和 server 不一致。**

接上面的例子：

| | Client A | Client B | Server |
|---|----------|----------|--------|
| push 前 | v5 + a.md 改动 | v5 + b.md 改动 | v5 |
| push 后 | v6（本地只有 a.md 改动） | v7（本地只有 b.md 改动） | v7（a.md + b.md 都有） |

- **Client A**：认为自己在 v6，但 server 已经到 v7 了。本地缺 B 的 `b.md` 改动。
- **Client B**：认为自己在 v7，但本地缺 A 的 `a.md` 改动 — 虽然 server 的 v7 包含 `a.md`。

如果 Client B 下次 pull（base=v7），server 说"v7 之后没有新变化" — 但 Client B 本地其实缺了 `a.md`！

**解决方案**：push 响应中返回合并后的变更集（merged changeset）。

```
Push 响应:
{
  "version": 7,
  "merged_changes": [        ← server 合并时额外引入的改动
    { "path": "a.md", ... }  ← 告诉 Client B: 除了你的改动，还有这些需要同步
  ]
}
```

Client 收到响应后：
1. 更新本地 base_version 到 v7
2. 把 `merged_changes` 中的文件写入本地 → 本地状态和 server 完全一致

这样 Client B 在 push 完成后就拥有完整的 v7 状态，不需要额外 pull。

Client A 则仍停在 v6（push 时没有其他人的改动需要合并，merged_changes 为空）。
A 下次 pull 时会正常拉到 v7 的变更（B 的 `b.md`）。这是预期行为，和 Git 一致。

### 3.5 合并策略

| 情况 | 处理方式 | 结果 |
|------|---------|------|
| A 改了 `a.md`，B 改了 `b.md` | 三方合并 | 两边都保留 |
| A 和 B 都改了 `readme.md` | 三方合并 + 服务端冲突解决 | 冲突部分的处理策略可配置（如 LWW、拒绝、标记冲突等） |
| A 删了 `old.md`，B 改了 `new.md` | 三方合并 | 删除和新改动都保留 |

### 3.6 各场景的理想行为

| 场景 | 并发行为 | 结果 | 等待时间 |
|------|---------|------|---------|
| S1/S3/S6: 改不同文件 | **并行** | 两边改动都保留 | ≈ 0 |
| S2/S4/S7: 改同一文件 | 并行计算 + 自动重试 | 服务端冲突解决 | ≈ 0（重试一次） |
| S5: 不相关 scope | **完全并行** | 各自独立 | 0 |
| S8/S9: 读取可见性 | 无需等待 | 嫁接保证可见 | 0 |

**核心指标**：不相关的写入（改不同文件、不同 scope）零等待。
Root scope 的存在不影响任何人的写入并发。
Push 涉及合并时，响应中返回 merged_changes，client 无需额外 pull 即可同步。

## 四、当前实现 vs 理想架构

下面逐一对比第三章的每个理想设计点，说明当前实现的差距、会在什么场景下出问题、以及后果是什么。

当前并发控制**基本不工作**。实际使用中没出问题是因为 push 操作很少真正并发。
一旦多个 Agent 并发写入同一项目，以下问题都会暴露。

### 4.1 对比 3.1 单一项目树

**理想**：更新 scope hash 时用 CAS — "只有 hash 没变才写入"，发现并发冲突则重试。

**现状**：更新 scope hash 时**无条件覆盖** — 不管中间有没有人改过，直接写入新值。

**后果**：两个 push 同时进行时，先完成的那个的改动会被后完成的直接覆盖，数据丢失。

| 场景 | 会发生什么 |
|------|----------|
| S1/S3: 同 scope 两个 push 改不同文件 | A 的改动被 B 的覆盖，`a.md` 的变更丢失 |
| S2/S4: 同 scope 两个 push 改同一文件 | 没有三方合并，后写直接覆盖先写 |

### 4.2 对比 3.2 乐观并发

**理想**：不加长锁，并行计算合并，最后 CAS 提交，失败自动重试。

**现状**：用的是悲观锁，但锁有三个问题导致**完全无效**：

**问题 A：每个请求的锁是独立的**

每次 push 请求都创建一个全新的锁对象。两个并发 push 各自拿各自的锁 — 永远不会互相阻塞。
就好比两个人各带了一把钥匙，各开各的锁，谁也挡不住谁。

**问题 B：锁按 AP 区分，不按 scope 路径区分**

两个不同的 AP 绑定了同一个 `docs/` 文件夹，但因为 AP ID 不同，拿到的是不同的锁。
即使修复了问题 A（让锁跨请求共享），不同 AP 之间的并发写入仍然不互斥。

**问题 C：拿不到锁直接拒绝**

锁被占用时直接返回 409 错误，要求客户端自己重试。
不是理想方案中的"server 自动重试，client 无感知"。
（因为问题 A 锁根本无效，这个问题目前没暴露。）

**后果**：所有并发写入场景（S1-S4、S6-S7）都不安全。
两个 push 可以同时执行完整的读-合并-写流程，后写覆盖先写。

### 4.3 对比 3.2 嵌套 scope 嫁接

**理想**：子 scope 提交成功后立刻返回 client，嫁接异步完成。
嫁接本身也用 CAS 保护，失败自动重试。父 scope 在毫秒级延迟内看到子 scope 的改动。

**现状**：嫁接确实在 push 返回后执行（方向对了），但有两个问题：

**问题 A：嫁接操作没有任何并发保护**

嫁接是"读取当前全局树 → 插入子 scope 的新树 → 写回"。
这个读-改-写操作没有 CAS 也没有锁。如果两个子 scope 同时嫁接，后者会覆盖前者的嫁接结果。

例如：`docs/` 和 `src/` 同时 push 后同时嫁接 → 只有后嫁接的那个 scope 的改动进入全局树，
另一个 scope 的嫁接被覆盖了。

**问题 B：父 scope 查询时绕过了全局树**

即使嫁接成功了，父 scope 在查询文件时优先查自己的 scope hash，不查全局树。
子 scope 的改动嫁接到了全局树上，但父 scope 不去看全局树 → 看不到子 scope 的改动。

**后果**：

| 场景 | 会发生什么 |
|------|----------|
| S8: 子 scope push 后，父 scope 读取 | 父 scope 看不到子 scope 的新文件 |
| S9: 父 scope push 后，子 scope 读取 | 子 scope 看不到父 scope 在自己范围内的改动 |
| S5: `docs/` 和 `src/` 同时 push | 两个嫁接竞争，可能丢失一个 scope 的嫁接结果 |

### 4.4 对比 3.3/3.4 Server 端流程 + Client 版本一致性

**理想**：CAS 失败后 server 自动重新读取最新状态、重新三方合并、重试提交。
Client 只发一次 push 请求。Push 响应包含 merged_changes，client 据此同步本地状态。

**现状**：

- **三方合并本身是有的** — 当 client 的 base_version 落后于 server 的 current_version 时，
  server 会执行三方合并。这部分逻辑正确。
- **但没有 CAS 重试** — 合并后直接写入，不检查中间有没有其他 push 改了 hash。
  如果两个 push 同时合并同时写入，后写覆盖先写，三方合并的结果被浪费。
- **push 响应不包含 merged_changes** — 只返回"是否发生了合并"和"冲突数量"，
  不返回具体合并引入了哪些文件。Client 无法知道 server 还合并了别人的什么改动。

**后果**：

| 场景 | 会发生什么 |
|------|----------|
| 两个 push 同时触发三方合并 | 两个合并结果都算出来了，但后写覆盖先写，先算的那个白算了 |
| Client B 的 push 合并了 A 的改动 | B 不知道 A 改了什么文件，B 的本地状态缺了 A 的改动，下次 pull 也拉不到（因为版本号已经是最新的了） |

### 4.5 对比 3.2 版本号

**理想**：版本号由 CAS 操作在数据库内原子递增，不可能重复。

**现状**：版本号是从数据库读出来 → 在内存里 +1 → 写回数据库。
两个并发请求可能同时读到同一个版本号，各自 +1，写出相同的版本号。

**后果**：版本历史中出现重复版本号，版本回溯和 diff 可能混乱。

### 4.6 各场景在当前实现下的实际行为

| 场景 | 理想行为（第二章） | 实际行为 | 后果 | 根因 |
|------|------------------|---------|------|------|
| S1/S3/S6: 改不同文件 | 并行，两边都保留 | 并行但无合并 | 先提交的改动丢失 | 4.1 无 CAS + 4.2 锁无效 |
| S2/S4/S7: 改同一文件 | 三方合并 + 冲突解决 | 无有效合并 | 数据丢失 | 4.1 无 CAS + 4.2 锁无效 |
| S5: 不相关 scope | 完全并行 | 并行 | **正确** | — |
| S8/S9: 读取可见性 | 可见 | 不可见 | 子 scope 改动对父 scope 隐身 | 4.3 嫁接无保护 + 查询绕过全局树 |

### 4.7 为什么现在没出问题

实际使用中 push 操作很少真正并发 — 人手动操作间隔足够大，单个 Agent 通常串行执行。
但一旦多个 Agent 并发写入同一项目（PuppyOne 的核心场景），以上问题都会暴露。

## 五、修复路线图

### Phase 1：保底安全（不丢数据）

**目标**：让并发 push 不再丢数据，子 scope 改动对父 scope 可见。

1. **修复锁**：让并发 push 真正互斥（共享锁、按 scope 路径而非 AP ID 加锁）
2. **阻塞等待**：拿不到锁时等待（带超时），不直接拒绝。
   第二个 push 等前一个完成后读到最新状态，三方合并自动保留不同文件的改动
3. **原子嫁接**：子 scope 写入时，嫁接到全局树作为 push 事务的一部分，而非事后异步
4. **修复版本号**：确保版本号全局唯一递增

| 场景 | Phase 1 后的行为 |
|------|-----------------|
| S1/S3/S6: 改不同文件 | 串行，三方合并保留两边 |
| S2/S4/S7: 改同一文件 | 串行，三方合并 + 冲突解决 |
| S5: 不相关 scope | 并行 |
| S8/S9: 读取可见性 | 可见（嫁接修复） |

**局限**：S1/S3/S6 中改不同文件也要串行等待（几秒）。
如果 root scope 可写，整个项目的所有写入都要排队（因为 root 和所有 scope 重叠）。

### Phase 2：乐观并发（核心目标）

**目标**：消除"改不同文件也要串行"的瓶颈。

1. 将写入流程改为 CAS 式（见 3.2 节）
2. CAS 只保护 scope 自身的 hash + 版本号写入（几毫秒）
3. 嫁接到父 scope 异步完成（见 3.2 "嵌套 scope 下的嫁接瓶颈"）
4. 合并计算过程不持锁，多个 push 可以并行计算

| 场景 | Phase 2 后的行为 |
|------|-----------------|
| S1/S3/S6: 改不同文件 | **并行**，自动合并 |
| S2/S4/S7: 改同一文件 | 并行计算 + 自动重试 + 冲突解决 |
| Root scope 可写 | **不阻塞其他 scope** |

**这是解决"root scope 阻塞一切"的根本方案。**

### Phase 3（可选）：文件级冲突感知

**目标**：减少不必要的重试。

CAS 失败时，对比两边修改的文件列表：
- 无交集 → 直接基于最新状态嫁接，无需重新合并
- 有交集 → 才需要重新三方合并

高并发场景下重试率大幅下降。

## 六、待讨论

1. **Phase 1 先上还是直接做 Phase 2？**
   Phase 1 改动小、上线快，但 root scope 可写时整个项目串行。
   Phase 2 是根本方案，但改动更大。
   建议 Phase 1 先上保底，Phase 2 跟进。还是直接做 Phase 2？

2. **Root scope 是否允许可写？**
   Phase 1 期间 root scope 可写会串行化所有写入，建议暂时限制为只读。
   Phase 2 后无需限制。团队是否接受这个临时约束？

3. **CAS 重试次数**：建议 3 次，超出返回错误让客户端重试。够吗？

4. **嫁接同步还是异步？**
   - 同步（push 事务内）：强一致，父 scope 立刻可见。但父 scope 行变成热点，所有子 scope 嫁接串行。
   - 异步（push 成功后异步嫁接）：写入强一致 + 可见性最终一致（毫秒级延迟）。子 scope 之间完全不阻塞。
   建议异步。团队能接受毫秒级的跨 scope 可见延迟吗？

5. **push 响应是否返回 merged_changes？**
   CAS 合并后 client 本地可能缺少其他 push 引入的文件改动。
   建议 push 响应包含 merged_changes，client 据此更新本地状态，无需额外 pull。
